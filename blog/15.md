## 前言

我的博客托管在GitHub上，自然在用git控制。

在博客的早期阶段，是直接把高清原图、音乐（MP3）存在根目录，自然也跟着源码一起到了GitHub。在前段时间我把图片迁移到了图床，并完全移除了音乐播放器，本以为万事大吉，但我忘了git仍然把这些文件保留进了历史。**整个git文件夹已经达到了120mb**，但博客本身的源码+目前的文章也仅仅500kb

虽然说清理收益不大，也许让cloudflare部署变快、也许让克隆变快、也许……

虽然收益不大，但也不能放着屎山不管，不能变成历史遗留问题。遂打算狠下心来移除git里面的垃圾文件（反正暑假时间多好折腾(ˉ▿ˉ)）

## 正文

**注意：此文章基于我自己维护的一个项目，如果存在多人维护请慎重**

根据我查到的，有两种方法

1. BFG Repo-Cleaner，简单好上手，但需要Java环境
2. git filter-repo，官方推荐，需要python环境

因为python环境我有现成的，我就用git filter-repo了

安装：
```shell
pip install git-filter-repo
```
安装完毕后，就可以开始清理了。我们可以先扫描git仓库来检查一下具体是什么大文件占用了空间

首先，如果你是Windows用户，**以下命令需要在Git Bash中运行**

**在操作之前，建议使用``git status``检测当然仓库是否有未push的提交**
```shell
git rev-list --objects --all | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | sort -k3 -n -r | head -20
```
参数简要解释如下：
* git rev-list --objects --all：列出所有提交中的文件对象（相当于把仓库里每一份文件都翻出来）。
* git cat-file --batch-check='...'：查询每个文件的大小、类型等信息。
* sort -k3 -n -r：按第3列（文件大小）进行数字倒序排列（最大的放最上面）。
* head -20：只显示前20行。

由输出可以看出来，排行榜前面的几乎全是音乐和图片。同时结合我们博客的自身结构进行分析，在博客早期壁纸文件直接放在根目录，以wallpaper开头，后面迁移到wallpaper文件夹。音乐则放在Music文件夹，因此，我们可以直接精确打击

```shell
git filter-repo \
  --path Music \
  --path wallpaper \
  --path-glob "wallpaper*" \
  --invert-paths \
```
参数简要解释如下：
* ``--path Music``：匹配 Music/ 文件夹（及其所有子文件）。
* ``--path wallpaper``：匹配 wallpaper/ 文件夹（及其所有子文件）。
* ``--path-glob "wallpaper*"``：匹配根目录下以wallpaper开头的文件
* ``--invert-paths``：反向选择。意思是“保留除上述路径以外的所有文件”，即删除这些路径。如果不加，则只保留这些路径（正好相反）。

如果你尝试直接对仓库进行操作，可能会报错
```shell
Aborting: Refusing to destructively overwrite repo history since
this does not look like a fresh clone.
  (expected freshly packed repo)
Please operate on a fresh clone instead.  If you want to proceed
anyway, use --force.
```

大概意思就是它检测到你的仓库不是一个全新克隆的干净仓库，所以拒绝直接运行，防止意外破坏。对于这个，你可以选择在刚才的命令加上``--force``参数，也可以现场``git clone``一次新仓库。这里我图方便选择使用``--force``参数强制清理。**但我强烈建议你们克隆个新仓库来操作，或者备份你的现有仓库**


清理完成后，我们需要重写GitHub上的git历史。首先，我们要重新添加远程仓库，因为filter-repo为了安全，自动移除了origin远程地址（防止你不小心推送错误的历史）。现在需要手动加回来
```shell
git remote add origin https://github.com/XUNDEI/xundei-blog.git
```
请将以上仓库替换为你的仓库

*如果你使用 SSH 地址，则换成 git@github.com:XUNDEI/xundei-blog.git*

接下来，强制推送新历史到GitHub
```shell
git push --force-with-lease origin main
```
**注意：如果你的默认分支是 master，请改为 master。**

推送过程中你会看到类似 + 361a360...xxxxx main -> main (forced update) 的提示，表示强制更新成功。（如果失败，考虑科学上网）

但是不出意外的还是出了了意外
```shell
To https://github.com/XUNDEI/xundei-blog.git
 ! [rejected]        main -> main (stale info)
error: failed to push some refs to 'https://github.com/XUNDEI/xundei-blog.git'
```
这个错误意思是，你的本地记录的远程分支信息已经过时，git认为远程可能有你本地不知道的新提交，所以拒绝强制推送（即使加了--force-with-lease也会基于过时的远程引用做检查，导致冲突）。

我本来打算尝试用``git fetch origin``先尝试获取远程最新信息，但更奇怪的报错出来了

这个错误是git在更新 pack 文件时，系统阻止它删除旧的.idx文件（不知道哪个神人程序锁定了）

不过问题不大，因为本地已经有远程分支的引用信息了（虽然对象没完全解包，但推送不依赖这些），输入n回车再次push即可。我这里直接力大砖飞用``--force``提交
```shell
git push --force origin main
```
（这么push有风险，因为我是单人开发就不怕，各位生产环境慎用）

注意到，push时git文件已变小
```shell
Writing objects: 100% (739/739), 2.08 MiB | 417.00 KiB/s, done.
```

从120MB降到2MB！完美！
